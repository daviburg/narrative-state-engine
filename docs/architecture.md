# Architecture

## Overview

The narrative-state-engine is a player-side assistant framework organized around a strict separation between **raw** (immutable) data and **derived** (reproducible) data.

```
External AI DM  <-->  Player  <-->  narrative-state-engine
```

The system never interacts with the AI DM directly. The player copies DM outputs and player prompts into the repository.

---

## Data Flow

```
DM response text
      |
      v
sessions/*/raw/full-transcript.md        (immutable append)
sessions/*/transcript/turn-NNN-dm.md     (immutable append)
      |
      +------------------------------------------+
      |                                          |
      v                                          v
tools/update_state.py                  tools/semantic_extraction.py  (optional, LLM)
      |                                          |
      +---> sessions/*/derived/state.json        +--- single-pass (default)
      +---> sessions/*/derived/objectives.json   |     +---> framework/catalogs/*.json
      +---> sessions/*/derived/evidence.json     |
      +---> sessions/*/derived/turn-summary.md   +--- segmented (--segment-size N)
      |                                          |     +---> segment 1..K catalogs (fresh)
      |                                          |     +---> reconcile_segments()
      |                                          |     +---> framework/catalogs/*.json
      |                                          |
      |                                          +---> framework/catalogs/events.json
      |                                          +---> framework/catalogs/timeline.json (temporal #263)
      v                                          |
      +------------------------------------------+
      |
      v
tools/derive_planning_layer.py       (catalog → derived planning files)
      |
      +---> sessions/*/derived/state.json        (populated from catalogs)
      +---> sessions/*/derived/evidence.json     (populated from events + entity attributes)
      +---> sessions/*/derived/timeline.json     (merged session + catalog timeline)
      |
      v
tools/build_context.py               (entity context for a specific turn)
      |
      +---> sessions/*/derived/turn-context.json
      |
      v
tools/analyze_next_move.py
      |
      +---> sessions/*/derived/next-move-analysis.md
      +---> sessions/*/derived/prompt-candidates.json
```

Semantic extraction is triggered automatically during `bootstrap_session.py` (batch)
or via the `--extract` flag on `ingest_turn.py` (incremental). It requires an LLM
endpoint configured in `config/llm.json` and gracefully degrades if unavailable.
The `--extract-only` flag on `ingest_turn.py` re-runs extraction against an existing
turn file (given via `--file`) without creating a new turn or modifying the raw
transcript, for re-extraction after a template or model change.

Planning layer derivation (`derive_planning_layer.py`) bridges the gap between
extracted catalog data and the derived planning files consumed by
`analyze_next_move.py`. It runs automatically when `update_state.py` is invoked
with `--framework`, or can be run standalone.

## Layer Descriptions

### Raw Layer (Immutable)

All original text is stored verbatim and never modified.

- `sessions/*/raw/full-transcript.md` — complete session transcript in order
- `sessions/*/transcript/turn-NNN-{player,dm}.md` — one file per turn

### Catalog Layer (Framework)

Running catalogs of entities, locations, factions, items, and plot threads extracted from all sessions.

- Updated after each turn
- Each entry includes `first_seen_turn` and `last_updated_turn` for traceability
- Catalogs grow over time; entries are never deleted
- A **relationship index** (`framework/catalogs/relationship-index.json`) is auto-generated on every `save_catalogs()` call, providing bidirectional lookup: given any entity ID, find all relationships where it is source (forward) or target (reverse). This enables efficient reverse queries like "what NPCs are at this location?" without scanning every entity file (#258)

### Scene Graph (Framework, Derived)

Cross-type spatial and temporal index built from entity catalogs (#257).

- `framework/catalogs/scene-graph.json` — generated by `tools/build_scene_graph.py`
- **Location index**: inverted mapping from location ID → entities present (via `volatile_state.location`)
- **Turn activity index**: mapping from turn ID → entity IDs introduced or updated on that turn
- **Location connections**: spatial edges between locations derived from location entity relationships
- Used by `build_context.py` for O(T) nearby-entity lookups via direct dict lookups into the turn activity index, instead of O(N) full-catalog scans
- Rebuilt from catalog files; no extraction rerun needed
- Falls back gracefully to full-scan when the index is absent

### State Layer (Derived, Per Session)

Per-session structured state derived from catalog data and transcript extraction.

- `state.json` — current world state, player state, constraints, opportunities, risks. Auto-populated from catalog entities (player location, condition, equipment, relationships), plot threads (active threads, opportunities), and entity attributes (known/inferred constraints, adversarial risks). Placeholder fields are replaced; manually authored content is preserved.
- `objectives.json` — current player objectives with status
- `evidence.json` — tagged evidence (explicit, inference, bait, hypothesis). Auto-populated from catalog events (→ explicit_evidence), entity attributes with inference flags (→ inference with confidence), and low-confidence relationships (→ inference). Existing entries are preserved; new entries are deduplicated before appending.
- `timeline.json` — merged session-level (pattern-extracted) and catalog-level temporal markers

### Analysis Layer (Derived, Per Turn)

Generated after each new DM turn.

- `turn-summary.md` — concise summary of the latest turn
- `next-move-analysis.md` — situation analysis answering: what changed, what's known vs inferred, bait, opportunities, risks, affected objectives
- `prompt-candidates.json` — multiple candidate prompts with rationale, risk, and objective alignment

### DM Profile (Framework)

A running profile of inferred DM behavior patterns, populated from two data sources:

1. **Transcript analysis** — `tools/dm_profile_analyzer.py` uses the LLM to extract behavioral patterns from DM turns (tone, structure, hints, adversariality, formatting). Runs in batch mode (via `bootstrap_session.py`) or per-turn (via `ingest_turn.py --extract`).
2. **User-provided documents** — Off-game notes, session-zero agreements, house rules passed via `--user-input`. Template at `templates/content/dm-profile-user-input.md`.

- Updated incrementally as patterns emerge
- Includes tone, structure, hint patterns, adversarial level, and formatting preferences
- All entries include confidence scores (capped at 0.9; 1.0 reserved for user-confirmed)
- Consumed by `tools/analyze_next_move.py` to inform risk assessment and prompt generation
- LLM prompt template at `templates/extraction/dm-profile-analyzer.md`

### Semantic Extraction Layer (Optional, LLM-based)

An automated pipeline that uses an LLM to extract structured data from transcript turns.

- **Five-agent pipeline**: Entity Discovery → Entity Detail → Relationship Mapper → Event Extractor → Temporal Signal Extractor
- **Intra-turn parallelism** (#282): When `parallel_workers > 1` in `config/llm.json`, entity detail, PC detail, relationship mapping, and event extraction run concurrently via `ThreadPoolExecutor` after discovery completes. Results are merged sequentially (entities → relationships → events) to maintain catalog consistency. The inter-call delay is applied once per turn instead of between each call. With `parallel_workers: 1` (default), the pipeline uses the original sequential execution path.
- **Multi-endpoint dispatch**: When `base_urls` (list) is configured in `config/llm.json`, `LLMClient` creates one `OpenAI` client per endpoint and distributes requests via thread-safe round-robin. This enables multi-GPU setups where each `ov_serve.py` instance targets a different GPU (e.g., `--device GPU.0` on port 8000, `--device GPU.1` on port 8001). With `parallel_workers > 1`, concurrent intra-turn requests are spread across all endpoints. Falls back to the single `base_url` when the list is absent.
- **Deterministic sampling + sampler observability** (#471): `LLMClient` reads optional `top_k`, `top_p`, `min_p`, and `seed` keys from `config/llm.json` (in addition to `temperature`) and threads them through every `extract_json()` call. `top_p`/`seed` are sent as native OpenAI parameters; `top_k`/`min_p` ride in `extra_body` (Ollama uses its own `options` mapping). Backend-dependent forwarding applies on the send path: the Ollama native streaming path (`_use_ollama_streaming`) forwards only `temperature`; the Ollama OpenAI-compat (`/v1`) path forwards `top_p`/`seed` but drops `top_k`/`min_p`; cloud OpenAI APIs forward `top_p`/`seed` but drop `top_k`/`min_p` (rejected as unknown params); self-hosted llama-server/vLLM accept all four. `_log_sampler_config()` annotates any configured-but-dropped sampler as `(not sent)` so the log never claims a sampler that did not reach the backend. All four default to `None` and are omitted from the request when unset, so existing configs are unaffected. The greedy/deterministic config is `temperature: 0, top_k: 1, top_p: 1.0, min_p: 0.0, seed: 42`; with a single pinned GPU endpoint this yields byte-deterministic extraction (the llama-server baked default is the *stochastic* `temperature 1.0`, so omitting these keys is not greedy). On startup, `_log_sampler_config()` writes the effective client-sent sampling to stderr (`[sampler]` prefix) and, for any non-cloud (self-hosted) backend (skipped under cloud providers and under `pytest`), probes the server's root `/props` endpoint and logs its `default_generation_settings` for independent verification. This includes self-hosted backends reachable via public hostnames when `self_hosted: true` is set in `config/llm.json`. The probe is best-effort: all network failures are swallowed and never raise. Cross-GPU `base_urls` round-robin remains the only determinism-breaking source once the deterministic config is in place. See `docs/ab-test-standard.md` §1.2/§1.4.
- Prompt templates in `templates/extraction/` define each agent's behavior
- `tools/semantic_extraction.py` orchestrates the pipeline
- **Kinship relationship prompting**: The relationship mapper template (`templates/extraction/relationship-mapper.md`) explicitly prompts for kinship relationships with family-tie indicators, bidirectionality guidance, confidence calibration, and examples. This addresses under-extraction of family bonds in tribal/community campaign settings.
- **Entity-detail / relationship-mapper responsibility split** (#505): Relationships are owned **solely** by the `relationship_mapper` phase, not `entity_detail`. The `entity-detail.md` template instructs the model to omit a `relationships` array and to omit the `notes` field when empty (never emit `null`). As a cause-independent net, the entity_detail response parser `_coerce_entity_fields()` drops any `relationships` (and relationship-variant keys such as `relations`, `character_relations`, `social_connections`) the model emits anyway, and drops `notes` when `null`. This eliminates two schema-validation failure modes that silently dropped the whole entity update: (a) `notes: null` (schema requires a string) and (b) echoed/invented relationships missing the required `first_seen_turn`. The `relationship_mapper` path (`_extract_relationships_parallel` → `merge_relationships`) is independent of `_coerce_entity_fields` and is unchanged, so real relationship data still flows through it.
- **Compound-term fragment rejection** (#398): A two-layer guard prevents extraction of single-word fragments of multi-word entity names. The programmatic layer: `_build_compound_word_index()` scans all catalog entities and same-turn discoveries at runtime, building a `{normalized_word → compound_name}` map from every word in every multi-word entity name (words are normalized with `re.sub(r"[^a-z]", "", word.lower())`). `_is_compound_term_fragment()` applies the same normalization to the candidate entity name before lookup, so fragments with trailing punctuation or mixed case are correctly matched. Entities whose single-word name appears in the index are rejected with reason `compound_term_fragment` and recorded in `discovery_filtered`. The prompt layer: `templates/extraction/entity-discovery.md` includes an ENTITY NAME VALIDATION section instructing the LLM to reject single-word fragments of compound terms (e.g., do not create `char-quiet` from "Quiet Weave") and treat single common English words as non-character names unless capitalized and directly addressed. The compound-term index is built dynamically from catalog data — no hardcoded word lists.
- `tools/catalog_merger.py` merges agent outputs into `framework/catalogs/`; includes per-pair relationship consolidation, `_dedup_relationships()` safety net (#183), and `cleanup_dangling_relationships()` to remove refs to non-existent entities (#184)
- `tools/llm_client.py` provides a provider-agnostic LLM wrapper (OpenAI, Ollama, Google Gemini, etc.)
- **Wall-clock watchdog** (#195, #281): Both the Ollama streaming path and the OpenAI-compat path enforce a hard wall-clock deadline of `timeout_seconds × 3`. If the LLM server stalls (GPU hang, network issue), the watchdog force-closes the connection and raises a retriable `LLMExtractionError`. This prevents indefinite hangs without affecting normal operation.
- `config/llm.json` configures the LLM provider, model, and endpoint
- All extracted entities are validated against `schemas/entity.schema.json` before merging
- Entities below a confidence threshold are logged but not cataloged
- Batch mode checkpoints progress every 25 turns (configurable via `config/llm.json` `checkpoint_interval`, default 25) for resume after interruption. At each checkpoint, progress is saved and entity catalogs and events are persisted to disk, so interrupted runs retain all entities discovered up to the last checkpoint.
- **Per-turn failure tracking** (#211): When entity discovery fails for a turn (e.g. HTTP 429, quota exhaustion, timeout), `extract_and_merge()` returns a `discovery_failed` flag. Failed turn IDs are accumulated during the batch and recorded in `extraction-progress.json` under a `failed_turns` array. A summary is printed to stderr after the extraction loop. On resume, previously failed turns are re-attempted before continuing with new turns. This prevents silent data loss where quota-exhausted turns were previously marked as successfully completed with zero output.
- **Structured extraction log** (#217, #250): Every call to `extract_and_merge()` returns a per-turn log record capturing the outcome of each extraction phase (discovery, detail, PC, relationships, events), any error messages, counts of new entities/events, and elapsed time. The record also includes `discovery_proposals` (every entity the model proposed, with `name`, `is_new`, `proposed_id`, `existing_id`, `confidence`) and `discovery_filtered` (entities rejected by confidence threshold, concept-prefix rules, or type misclassification filters, with rejection reasons). In batch, segmented, and single-turn modes the record is appended to `<framework-dir>/extraction-log.jsonl` (one JSON object per line). The file is append-only and written after each turn, so it survives interruptions. It immediately answers which turns failed, what error each phase produced, what the model proposed vs. what was accepted, and when failures started. Skipped in `dry_run` mode.
- **Event type coercion** (#198): Before schema validation, `_coerce_event_fields()` remaps invalid event `type` values to the nearest valid enum value using `_EVENT_TYPE_REMAP`. For example, `"acquisition"` → `"discovery"`, `"conflict"` → `"encounter"`. Unknown types fall back to `"other"`. This prevents valid events from being silently dropped when the LLM produces a type not in the schema enum.
- **entity_detail validation observability** (#503): When an `entity_detail` completion fails `_validate_entity` (schema validation), the offending update is dropped. Historically that drop was stderr-only, so per-turn metrics were blind to it. Every entity_detail validation site (solo, char-player, batched, backfill, refresh) now routes through `_validate_entity_detail()`, which records a structured failure — `{turn, entity_id, phase, error}` where `error` is `<json-path>: <message>` (e.g. `['notes']: None is not of type 'string'`) — into a per-call buffer. The buffer is held in a `contextvars.ContextVar` (not a process-global list) so each `extract_and_merge()` / `backfill_stubs()` / `refresh_entities()` call gets its own buffer; this keeps recording correct when those calls run concurrently (e.g. `tools/retry_failed_turns.py` drives them through a `ThreadPoolExecutor`), where a shared global would let one turn drain or mis-attribute another turn's failures. The detail phase fans out across an inner `ThreadPoolExecutor`, so those submissions copy the call's context (`_submit_in_context`) to share the one buffer with their workers. For the per-turn detail phases, `extract_and_merge()` drains the buffer into a `validation_failures` list on the turn's log record (the key is omitted, not null, when there are no failures, so a clean turn's record stays byte-identical to the baseline). A fresh buffer is installed at the start of each turn so failures never leak across turns or across reused worker threads. The post-turn passes (backfill, refresh) build no per-turn record, so they flush their failures via `_flush_validation_records_to_log()` as a standalone `extraction-log.jsonl` record that carries no top-level `turn_id` and no `discovery_ok` flag — every log consumer (resume, retry, A/B paired scorer, compression aggregator) safely ignores it, while the originating turn is preserved inside each `validation_failures` entry. This is measurement-only: it does not change whether an entity is dropped or merged (the recovery behaviour is #504).
- **entity_detail sanitize + recover** (#504): A single recoverable schema violation (e.g. `notes: null`, or one relationship missing `first_seen_turn`) previously made `_validate_entity` reject the WHOLE entity, so `merge_entity` was skipped and the entire turn's update for that entity was lost. `_validate_entity_detail()` now runs a schema-driven pre-validate sanitizer, `_sanitize_entity_for_validation()`, before validating. The sanitizer reads optional/required keys and types from `entity.schema.json` (Rule 9 / Rule 10 — **no** hardcoded field or word lists) and strips ONLY the specific invalid pieces in place: (i) it drops null-valued **optional** string-typed top-level keys (e.g. omits `notes: null`); and (ii) for **every** top-level array property whose schema `items` declare required sub-fields (discovered generically from the schema — `relationships` is one such property, but the field name is **not** hardcoded), it drops array entries missing a **required** sub-field (or with one set to null), keeping the valid entries. After sanitizing, the entity is re-validated; if it now passes, its valid remainder merges. Unrepairable violations — a missing/wrong-typed **required** top-level field, an extra property, a bad enum value — are left intact so validation still rejects them (recorded per #503); the recovery never force-merges garbage and never mutates a value elsewhere. Relationships continue to flow through the relationship-mapper phase; the sanitizer only prevents one bad array entry from sinking the whole entity update.
- **entity_detail recovery logging** (#504): The sanitizer above REPAIRS recoverable violations *before* validation, so a recovered entity never reaches `_record_validation_failure` — the repair would otherwise leave **no** audit trail (a silently-wrong repair would be undetectable, the same anti-pattern as silent loss that #503 exists to surface). `_sanitize_entity_for_validation()` therefore takes an optional `repairs` out-param and reports WHAT it repaired — a schema-derived `repair` action plus the json-`path` (e.g. `{repair: "dropped null optional-string key", path: "notes"}`), and for an incomplete array item the dropped entry's index path plus the `missing` sub-field (e.g. `{repair: "dropped incomplete array item missing required subfield", path: "relationships[1]", missing: "first_seen_turn"}`). `_validate_entity_detail()` stamps each repair with `{turn, entity_id, phase}` via `_record_validation_repairs()` into a **second, parallel** per-call buffer — its own `contextvars.ContextVar` + lock, reset (`_reset_validation_repairs()`) and drained alongside the `validation_failures` buffer at every entry point and shared with inner-`ThreadPoolExecutor` workers through the same `_submit_in_context` context copy, so concurrent calls cannot drain, lose, or mis-attribute each other's repairs. The per-turn detail phases attach them as a `validation_repairs` list on the turn's log record; the post-turn passes (backfill, refresh) flush them via `_flush_validation_records_to_log()` in the same standalone record. The key is **distinct** from `validation_failures` (repairs = recovered-and-merged; failures = unrepairable-and-still-dropped) and is omitted, not null, when nothing was repaired, so a turn with no repairs stays byte-identical to the baseline. This is measurement-only: it only logs repairs that already happen — it never changes WHICH pieces the sanitizer strips or which entities merge (proven byte-identical by sanitizing with vs. without the `repairs` out-param). Note that `_coerce_entity_fields` (#505) runs *before* the sanitizer at every call site and independently removes `notes: null` and strips relationship keys (relationships are owned by the relationship_mapper phase), so at the call sites those two specific repairs are handled upstream by coerce and the `validation_repairs` channel surfaces the sanitizer's *remaining* recoveries (e.g. a `current_status: null` optional-string drop, or an incomplete required-item array entry). Routing those coerce-side strips through the same `validation_repairs` channel is a recommended follow-up.
- **Segmented extraction** (`--segment-size N`): For long sessions (300+ turns), extraction runs in segments of N turns, each starting with a fresh catalog. This prevents context window saturation in the entity discovery prompt, which includes the full known-entities table. After all segments complete, a reconciliation pass merges entities by ID and name, stitches event timelines, and joins relationships across segment boundaries. Segment size should be tuned to the model's effective context capacity (recommended: 100-150 for 14B models with 32K context). Entity catalogs are persisted after each completed segment and at intra-segment checkpoint intervals, so interrupted segmented runs also retain recoverable entity data (#220).
- **Incremental extraction** (`--start-turn N`, #251): Extract a specific range of turns using existing catalogs as prior context. Combined with `--max-turns`, which becomes an absolute turn number (upper bound) when `--start-turn` is set. E.g. `--start-turn 26 --max-turns 50` extracts turns 26–50. Wiki pages are auto-generated after each increment for human review. The tool prints a suggested next-batch command after completion.
- **Discovery temperature** (#251): The `discovery_temperature` key in `config/llm.json` sets a separate LLM temperature for the entity discovery phase. Higher values (e.g. 0.3) can improve entity recall at the cost of precision. When omitted, discovery uses the global `temperature`. The `LLMClient.extract_json()` method accepts a `temperature` parameter to override the default per-call.
- **Type classification guidance and filter** (#303): The entity discovery template includes a compact type classification guide that defines each entity type with positive examples and explicit NEVER-use rules. A post-discovery programmatic filter (`_is_misclassified_character`, `_is_misclassified_location`) rejects entities that match known non-sentient name patterns (e.g. diseases, body parts, events classified as "character") or non-spatial patterns classified as "location". Rejections are logged to stderr and recorded in `discovery_filtered` with reason `misclassified_character` or `misclassified_location`.
- **Compact discovery output** (#310): The entity discovery template defines a two-tier output format — new or changed entities use the full 7-field format, while known entities with no new information use a compact 2-field format (`{"existing_id": "...", "confidence": 0.9}`). After discovery, `extract_and_merge()` expands compact entries by looking up `name` and `type` from the existing catalogs, ensuring downstream code (detail selection, relationship builder) works without modification. Models that ignore the compact format and emit full entries for all entities are handled transparently — the expansion is a no-op when `name` is already present.
- **Bounded entity context** (#221, #310): The entity discovery prompt uses `format_known_entities_bounded()` to limit the known-entity table to a configurable token budget. Entities use a **three-tier temporal format**: entities active within a recent window (default 10 turns) or in a priority tier (mentioned, co-located, one-hop) get **full** detail (ID, name, type, identity, aliases); non-priority entities aged 11–20 turns get **brief** format (ID, name, type); non-priority entities aged 21–50 turns get **id-only** format (ID, name only). Entities that exceed the budget are omitted with a truncation note. Under budget pressure, degradation proceeds in three passes: full→brief, brief→id-only, then omission from the tail. The budget defaults to 25% of `context_length` from `config/llm.json` and can be overridden with the `entity_context_budget` key. Large-context models receive the full entity list when it fits within budget.
- **Context-aware entity selection** (#233): When the current turn's text is available, `format_known_entities_bounded()` uses a multi-tier selection strategy instead of recency-only ordering. Priority tiers: (1) **Mentioned** — entities whose name or alias appears in the turn text (word-boundary-aware matching, names shorter than 3 characters skipped); (2) **Co-located** — entities sharing a `volatile_state.location` with a mentioned entity, or entities at a mentioned location (matches both location names and `loc-*` IDs); (3) **One-hop relationships** — *active* `relationships[].target_id` targets of mentioned entities (dormant/resolved relationships are skipped); (4) **Recency backfill** — remaining entities sorted by `last_updated_turn` descending. Within each tier, entities are sorted by recency. Mentioned, co-located, and one-hop entities receive full detail regardless of recency; backfill entities use the same recency-window logic as before. **Proactive staleness trimming**: when `turn_text` is provided, the fast-path (return unbounded output when it fits the budget) is bypassed, and backfill entities whose `last_updated_turn` is more than 50 turns behind `current_turn` are excluded entirely — even when the token budget would accommodate them. This prevents stale, irrelevant entities from polluting the discovery prompt. Priority entities (mentioned, co-located, one-hop) are never excluded by staleness. When `turn_text` is not provided, the function falls back to the pre-#233 recency-only behavior with no staleness filtering. **False-positive mention filtering** (#297): names matching a common-word blocklist (e.g. "echo", "field", "fire") are excluded from mention detection when they are single-word names, to prevent extraction-artifact entities from triggering false matches. Multi-word names containing blocklisted words still match normally. **One-hop cascade cap** (#297): when the combined set of mentioned + co-located + one-hop entities exceeds 50% of the catalog (and the catalog has 20+ entities), one-hop expansion is skipped entirely to prevent false-positive cascade.
- Birth events trigger automatic entity creation for named children, with child IDs added to event `related_entities`
- Stub backfill gathers context from both `related_entities` references and entity name mentions in event descriptions
- **Type-specific orphan thresholds** (#185): The post-batch orphan sweep uses different minimum-reference thresholds by entity type — characters require 3 event references, locations require 2, and factions require 1. This improves late-game entity discovery for location and faction entities that appear less frequently in events.
- **Turn-tag ID normalization** (#185): Entity IDs with LLM-generated turn-tag suffixes (e.g., `char-shaman-turn-082`) are normalized to their canonical form (`char-shaman`) during orphan stub creation, post-batch orphan sweep, and the dedup pre-pass. The `_normalize_entity_id()` function strips `-turn-NNN` suffixes when the canonical form already exists in catalogs or the tagged form is unknown.
- **Name-mention discovery** (#185): After the post-batch orphan sweep, `_name_mention_discovery()` scans all event descriptions for capitalized proper names that do not match any known entity. Names appearing in 2+ distinct events get a character stub. This catches named characters like Borin and Elder Lyra that the LLM failed to include in `related_entities` arrays, closing the gap where entities are mentioned narratively but never formally extracted.
- **Stub character filter** (#384): Before creating character stubs in orphan sweep, name-mention discovery, and per-turn stub creation, _is_plausible_character_name() rejects implausible names using two dynamic strategies: (1) **Morphological suffix patterns** — words ending in common English suffixes (-tion, -sion, -ness, -ment, -ance, -ence, -ity, -ular, -ical, -ive, -ous, -ful, -less, -ern) are rejected as abstract nouns or adjectives. Proper names (capitalized, >6 chars like Laurence, Constance) bypass this check. (2) **Word frequency analysis** via the optional wordfreq library — words with English corpus frequency ≥ 3×10⁻⁶ are rejected as common English words. When wordfreq is not installed, only morphological patterns are used and a warning is logged at import time. Multi-word names and names in _GENERIC_STEMS are handled separately upstream.
- **Stale-item sweep** (#367): After name-mention discovery, `_sweep_stale_items()` removes item entities with insufficient event references after a staleness window. An item is pruned when: (1) it has fewer than `stale_item_min_refs` event references (default: 2), and (2) at least `stale_item_window` turns (default: 25) have passed since its `first_seen_turn`. Only items are swept — characters, locations, and factions are never removed. This reduces catalog noise from ephemeral scene props (bowls, utensils, water) that have no narrative significance. Runs in both non-segmented and segmented extraction paths (post-reconciliation).
- **Season coercion** (#185): `_normalize_season()` in `extract_structured_data.py` maps LLM season output variants (hyphenated, compound, colloquial) to schema-compliant enum values. The `state.schema.json` season enum includes sub-season variants: `early_spring`, `late_summer`, `mid_winter`, etc.
- **Periodic entity refresh** (#161, #182): Every `entity_refresh_interval` turns (default 50), the pipeline identifies entities whose `last_updated_turn` has fallen behind by more than the interval. Up to `entity_refresh_batch_size` (default 10) of the most stale entities are re-extracted using recent transcript context where they are mentioned. This prevents entities from going permanently stale in long sessions. The refresh uses the same entity-detail LLM template and merges results via `merge_entity()`. **Dynamic scaling** (#182): when the catalog has 60+ entities, the effective batch size scales to `max(batch_size, catalog_size // 5)`, capped at 25. **Type-aware slot allocation** (#182): refresh slots are distributed proportionally by entity type — characters 50%, locations 20%, items 20%, factions 10% — so narratively important characters are refreshed more frequently. Unused type slots overflow to other types. When an events list is available, entities referenced in more events win ties between entities with similar staleness gaps. **End-of-run refresh** (#212): After the main extraction loop completes (both segmented and non-segmented paths), a final refresh pass runs if the last turn does not land on a modulo boundary. This catches entities that went stale between the last periodic refresh and the end of the run. The final pass uses a larger batch size — the greater of the configured `entity_refresh_batch_size` and the capped maximum batch size of 25 — to handle the potentially larger backlog of stale entities.
- **Entity detail envelope handling** (#168): Entity detail extraction accepts both the standard `{"entity": {...}}` envelope format and flat entity responses where the LLM omits the wrapper. The `_unwrap_entity_response()` helper normalizes both formats, preventing valid entity data from being silently dropped with smaller models.
- **Non-standard key coercion** (#170, #172): `_coerce_entity_fields()` remaps common non-standard top-level keys returned by smaller LLMs into their correct V2 schema locations before validation. Volatile remaps include `equipment`/`inventory`/`equipment_and_tools`/`item_equipment`/`item_inventory` → `volatile_state.equipment`, `location`/`current_location` → `volatile_state.location`, `status`/`health_status`/`status_effects` → `volatile_state.condition`. Stable remaps include `abilities`/`skills_and_abilities` → `stable_attributes.abilities`, `alignment` → `stable_attributes.alignment`, `weaknesses` → `stable_attributes.weaknesses`. Relationship variants (`relations`, `character_relations`, `faction_relations`, `item_relations`, `items_relations`, `current_relationships`) are adopted into `relationships`. Known noise keys (26+, including `events`, `activities`, `activity_history`, ephemeral per-turn data) are discarded. A `_new` suffix strip pass normalizes diff-format keys like `relationships_new` → `relationships` before remap/discard runs. Null values in `stable_attributes.*.value` are stripped before validation (#178).
- **Location ID normalization** (#322): After entity detail extraction, `_normalize_entity_location()` replaces free-text `volatile_state.location` values with `loc-*` entity IDs when the text matches a known location's name or alias (case-insensitive). This enables reliable spatial queries via the scene graph builder and co-location filtering. The entity detail template also instructs the LLM to use `loc-*` IDs directly. Normalization runs at all 6 `merge_entity()` call sites (parallel detail, PC extraction, non-parallel detail, stub backfill, entity refresh).
- **PC extraction handling**: The player character (char-player) is extracted every turn with an extended timeout and a separate `pc_max_tokens` limit (configurable in `config/llm.json`) to accommodate its larger context. If PC extraction fails for 20 consecutive turns (`_PC_SKIP_THRESHOLD`), it enters a cooldown cycle: skipping 50 turns, then retrying for 5 turns, repeating until a success resets the counter. This replaces the previous permanent-skip behavior so the PC can recover from transient failure streaks.
- **Rate limit handling** (#215): The LLM client (`tools/llm_client.py`) tracks per-request error statistics via `RetryStats`, classifying errors by HTTP status code and detecting `Retry-After` headers. SDK-level retries are disabled (`max_retries=0`) to prevent the OpenAI client's default 2 retries from multiplying with the pipeline's `retry_attempts` (previously causing up to 9 requests per call). Backoff uses the server's `Retry-After` value when available (≤60s), otherwise exponential backoff with jitter for cloud providers. **Quota exhaustion detection**: When consecutive 429 errors reach `consecutive_rate_limit_threshold` (default 10, configurable in `config/llm.json`), a `QuotaExhaustedError` is raised. This propagates through `extract_and_merge()` to the batch/segmented extraction loop, which saves progress and stops rather than burning through retries against a daily quota wall. `RESOURCE_EXHAUSTED` error messages are treated as 429 equivalents. **Cloud provider delay**: For non-local providers, `delay()` enforces a minimum 2000ms inter-call delay regardless of `batch_delay_ms` config. **End-of-run stats**: Both batch and segmented extraction print an API statistics summary (total requests, successes, error breakdown, Retry-After headers seen, max consecutive rate limits).
- **Fallback LLM provider**: When the primary provider exhausts all `retry_attempts`, the client automatically routes the request to a fallback provider configured via the `"fallback"` block in `config/llm.json`. The fallback block accepts the same keys as the top-level config (e.g. `base_url`, `model`, `timeout_seconds`, `skip_response_format`). The fallback client is only initialized when no `overrides` are passed (preventing recursive fallback creation). If both primary and fallback fail, the error message includes diagnostics from both providers. `LLMTruncationError` and `QuotaExhaustedError` bypass the fallback and propagate immediately.
- **PC alias merge** (`_merge_pc_aliases`): Post-extraction pass that identifies character entities which are actually aliases of the player character (e.g., the PC's proper name appearing as a separate entity). Candidates must appear ≥2 times in PC event text within a ≤3 turn span. Seven guards prevent false positives: (1) **co-occurrence guard** — skips merge if the candidate and char-player both appear in the same event's `related_entities`, indicating distinct characters; (2) **relationship guard** — skips merge if either entity has a relationship targeting the other; (3) **alias blocklist** (#186) — rejects meta-labels like "player character", "pc", "protagonist" via `_PC_ALIAS_BLOCKLIST`, and strips any existing blocklisted aliases from the PC entity; (4) **word blocklist** (#239) — rejects names matching `_PC_ALIAS_WORD_BLOCKLIST` (common English words, abstract nouns, NPC role descriptors like "healer", "warrior", "elder"); (5) **title prefix guard** (#239) — rejects names starting with "The " (e.g., "The Elder"), which indicate NPC titles rather than PC aliases; (6) **non-PC relationship guard** (#239) — skips merge if the candidate has ≥2 relationships targeting non-PC entities, indicating an independent character with its own relationship graph; (7) **independent event guard** (#239) — skips merge if the candidate appears in ≥2 events where char-player is not referenced, indicating independent narrative existence. Additionally (#214), aliases are validated by `_filter_pc_aliases()` which rejects common English words (`_PC_ALIAS_WORD_BLOCKLIST`), aliases shorter than 3 characters, and caps the alias list at 10 entries. Additionally (#302), `_filter_pc_aliases()` now cross-references aliases against all entity names in the catalog, rejecting any alias that matches another entity's primary name (case-insensitive). This prevents the LLM from contaminating an entity's alias list with names of distinct NPCs. A parallel guard `_filter_entity_aliases()` applies the same cross-reference check to non-PC entities during the stable_attributes merge in `_update_existing_entity()`.
- **PC volatile_state pruning** (#214): After each PC merge (both full and partial), `_prune_pc_volatile_state()` caps `volatile_state` keys at 30 (`_PC_VOLATILE_STATE_MAX_KEYS`). Core keys (`condition`, `equipment`, `location`, `current_activity`, `last_updated_turn`) are always preserved; excess non-core keys are dropped oldest-first (by insertion order). The entity-detail template also instructs the model to reuse existing keys and prefer `current_activity` over creating new per-turn keys.
- Biography sections use LLM-generated descriptive titles (not generic "Phase" labels), cached in `.synthesis.json` sidecars
- Wiki pages include cross-page entity links: the first mention of each known entity in biography prose, relationship tables, event timelines, and member/connection lists is a clickable markdown link to that entity's wiki page. Link resolution uses relative paths across entity types.
- **Coreference hints** (`apply_coreference_hints`): Optional manual merge rules in `sessions/*/coreference-hints.json`. Each entry maps a canonical entity ID to variant names and ID patterns. After the automatic dedup pass, the pipeline loads any hints file from the session directory and deterministically merges variant entities into their canonical counterpart — absorbing relationships, events, and stable attributes, deleting variant files, and rewriting all dangling references. Validated against `schemas/coreference-hints.schema.json`.
- **Dedup audit** (`tools/dedup_audit.py`): Post-extraction tool that identifies duplicate entities via name similarity heuristics (Levenshtein distance ≤3, substring match, same-turn co-occurrence), scores candidate pairs with an LLM call (confidence 0–1) based on name similarity, description overlap, relationships, and source turns, and either auto-merges high-confidence duplicates (≥0.9) via coreference hints or flags medium-confidence pairs (0.6–0.9) for human review in `dedup-review.json`. Reviewed pairs can be applied via `--apply-review`. Pairs where both entities have ≥3 relationships to distinct targets are excluded from candidacy to avoid false positives.
- **Periodic dedup audit** (#366): During extraction, `_run_periodic_dedup()` runs every `dedup_audit_interval` turns (default: 50, configurable in `config/llm.json`, set to 0 to disable). It reuses `dedup_audit.generate_candidates()` to find candidate pairs from the in-memory catalog state, scores them with an enhanced LLM prompt that includes narrative evolution awareness (recognizing entities that were renamed, rebuilt, or upgraded over time as the same entity), auto-merges high-confidence pairs (≥0.9) with the name-mismatch guard bypassed (since the LLM has already confirmed entity identity), rewrites event references and relationship IDs, and logs results. **Current limitation:** candidate generation uses string-similarity heuristics only (Levenshtein distance, substring match, same-turn co-occurrence), so entities with completely different names will not be surfaced as candidates. Entities with partial name overlap are covered. Runs in both segmented and non-segmented extraction modes, at the same hook point as entity refresh.
- **Same-turn dedup gate** (#337): The entity-discovery template instructs the LLM to avoid proposing multiple synonym entities for the same concept within a single turn (e.g. "safe haven", "hidden refuge", "defensible sanctuary" should produce one entity, not three). This is enforced at the template/prompt level — the extraction template contains explicit instructions to prefer the most specific name when multiple synonyms appear. A runtime `_same_turn_dedup_gate()` function is planned but not yet implemented.
- **Temporal extraction integration** (#263): The semantic extraction pipeline calls `temporal_extraction.extract_temporal_signals()` per-turn after event extraction (Phase 5). Pattern-based detection identifies season transitions, time-skip language, biological markers, and construction milestones directly from turn text and finalized events. Signals are merged into the timeline via `merge_temporal_signals()` with deduplication by `(source_turn, type, season, raw_text)`. The timeline is loaded alongside catalogs and events, saved at every checkpoint and at the end of extraction, and reconciled across segments in segmented mode. The extraction log records `temporal_ok`, `temporal_error`, and `new_temporal_signals` per turn. When `timeline` is not passed to `extract_and_merge()` (legacy callers), temporal extraction is skipped gracefully.

### Context Budget Optimizations (Extraction)

The extraction pipeline applies context budget controls automatically to prevent prompt overflow as catalogs grow. All optimizations are always active — no configuration needed.

**Discovery** (always active): Uses 4-tier priority scoring (mentioned → co-located → one-hop → recency backfill) with a token budget (25% of context window) and automatic tier degradation — full → brief → id-only → omit. See #233, #310.

**Relationship Relevance Scoring**: Applies a 3-tier priority system to `_collect_existing_relationships()`:
- Tier 1 (full history): both endpoints mentioned in current turn
- Tier 2 (current + last update): one endpoint mentioned + updated within 15 turns
- Tier 3 (summary only): one endpoint mentioned + active status
- Omit: dormant/resolved relationships unless both endpoints are mentioned
- Token budget: 20% of context window with automatic tier degradation

**Arc-Aware Compression**: Generalizes the PC-only volatile digest and relationship compaction to all entities:
- History arrays capped to 3 entries per key (same as PC)
- Entries older than 50 turns digested to a summary line
- Relationship histories trimmed to last 3 entries (arc summaries used for PC when available)

**Scene-Scoped Entity Detail**: Trims non-PC catalog entries in the entity-detail prompt:
- Volatile state: digested + capped to 3 entries per key
- Relationships: filtered to mentioned + recent (10 turns), capped at 8
- Stable attributes: preserved in full
- Entity detail calls per turn capped at 6 (PC excluded from cap; new entities prioritized over existing)

**PC Type-Tiered Relationship Cap** (L1+L4, epic #477): Closes the gap left by #385, which capped non-PC relationship detail and the relationship-mapper budget but left the PC path uncapped. Gated via `context_optimizations.relationship_type_tiering` in `config/llm.json` (**default ON** — the 150t A/B measured the flag-ON arm as merge-ready at quality parity with the per-call PC-context slope roughly halved, so the default was flipped from OFF to ON; the new 344t full-context baseline extraction runs on this default-ON state). Set the flag `false` to restore pre-#477 behaviour, which stays byte-identical to the pre-#477 baseline (main before #477; after this default-ON change merges, main itself ships the flag ON, so the escape hatch is parity with the pre-#477 baseline, not with current main) — the same-commit flag-OFF run serves as the A/B control.
- **L1 (entity_detail PC path)**: `_format_prior_entity_context` applies `_apply_pc_rel_type_tier()` to the PC relationship list. Relationship types are partitioned into two groups:
  - *Permanent bonds* (`kinship`, `adversarial`, `mentorship`, `political`, `partnership`): kept uncapped — these are long-arc callback anchors that never resolve in practice and cannot be reliably filtered by recency or status. The default list is a subset of the relationship `type` enum in `schemas/entity.schema.json` (asserted by a unit test); operators can override it with custom types via `pc_rel_permanent_types`.
  - *Volatile tail* (all other schema types — `factional`, `social`, `romantic`, `spatial`, `other`): capped at `pc_rel_volatile_tail_cap` (default 10), sorted by recency (most recent first).
  - *Mentioned-this-turn force-keep*: any relationship whose target appears in the current turn's mentioned-entity set is kept regardless of type or cap position.
  - Result: a 109-relationship PC web shrinks to ~30–40 retained entries without dropping any permanent-bond type or any mentioned-this-turn relationship. A/B validation (344t, same-commit flag-OFF control, port-8000 lane, P1) is a separate follow-up task after this PR merges.
- **L4 (relationship_mapper budget)**: `_collect_existing_relationships` applies the same permanent-type logic to the tier-assignment step, with two distinct guarantees that depend on whether an endpoint is mentioned this turn:
  - *Permanent bond with one endpoint mentioned*: promoted to at least tier 2. Tier 2 survives the first budget-degradation pass in `_format_relationships_budgeted` (which only degrades tier 3 → omit); under further pressure it is degraded to a tier-3 summary but is still emitted, not omitted. This is the strong guarantee — a mentioned permanent bond survives budget degradation.
  - *Permanent bond with neither endpoint mentioned*: the tier 4 → 3 boost keeps it as a summary **only when the block is within budget**. It is **not** an unconditional survival guarantee: the first budget-degradation pass degrades tier 3 → omit, which reverses the boost, so an unmentioned permanent bond can still be dropped under budget pressure. The boost simply makes the volatile tail (which also degrades) compete on equal footing rather than guaranteeing the bond outlives the budget.

  Net effect: the volatile tail is trimmed first, mentioned permanent bonds always survive, and unmentioned permanent bonds survive as summaries when budget allows. Configurable via the same `relationship_type_tiering` flag.
- Permanent-type list and volatile-tail cap are configurable in `config/llm.json` under `context_optimizations` (`pc_rel_permanent_types`, `pc_rel_volatile_tail_cap`). These are structural schema-type enums, not domain content words (Rule 9).

**Batched Entity Detail** (L2, epic #477, #491): Collapses the dominant late-turn token cost — the fixed entity-detail system template (~1,644 tok) plus the full turn text (grows to ~1,143 tok) being re-sent on *every* per-entity detail call (~6/turn). When ON, those shared parts are sent **once per batch** and multiple entities are detailed in a single shared-context call. Gated via `context_optimizations.batch_entity_detail` in `config/llm.json` (**default OFF** — flag-OFF is byte-identical to the per-entity baseline, which is the paired-A/B control). When the flag is absent or `false`, `_partition_detail_tasks()` returns every task as a solo task with no batch groups, so the per-entity code path is unchanged.

- **Safety tiering (anti attribute-bleed)**: `_partition_detail_tasks()` keeps the player character (`char-player`), brand-new entities, and entities with `confidence >= high_confidence_threshold` (default `0.8`) as **SOLO** calls; only the lower-salience tail (existing, lower-confidence entities) is batched, in groups of `batch_size` (default `4`). A residual group of a single entity is demoted to solo. `batch_size` and `high_confidence_threshold` are config-driven, not magic constants (Rule 10): the default `4` balances the token saving against cross-entity attribute-bleed risk and per-call output growth.
- **Prompt structure**: `format_detail_batch_prompt()` emits the turn header once, then **hard-delimited** per-entity blocks (`===== ENTITY BLOCK N START/END =====`). Each block carries the **same** per-entity prior context (`_format_prior_entity_context`) the solo path would send, so batching changes only the call grouping and prompt structure — not the extraction semantics (same entities detailed, same fields). The batch uses a dedicated `templates/extraction/entity-detail-batch.md` system template that returns `{"entities": [...]}`.
- **Robust split + anti-bleed fallback**: `_parse_batch_detail_response()` splits the batched response back to per-entity results by `id` (accepting the `{"entities": [...]}` envelope, a bare list, or a single-entity envelope) and **fails closed on duplicate ids** — an id repeated across objects is ambiguous and a cross-entity bleed vector, so it is dropped entirely and re-extracted solo. `_extract_batched_entity_detail()` enforces structural integrity to prevent silent attribute bleed: it ignores ids that were never requested and rejects any returned object whose `name`/`type` does not match its requested block (`_batch_entity_consistent()`), guarding against a misattributed/swapped body returned under the wrong id. It falls back to a **per-entity solo call** for any entity missing from, invalid in, mismatched in, or duplicated in the batched response, and for a whole-batch parse/LLM failure — so no entity is ever dropped, merged away, or silently misattributed. Both the sequential and `parallel_workers > 1` paths consume the batched results uniformly.
- **Non-blind measurement (the A0 lesson)**: the recorded `entity_detail` metric must reflect the prompt L2 actually changes. The per-turn instrumentation records the **actual batched prompt** (both the compacted-sent `input_tokens` and the #484 uncompacted `raw_input_tokens`) once per group, so the per-turn `raw_input_tokens` (a sum across calls) drops by ~`(K-1) × (template + turn_text)` for a batched turn of `K` entities. Crucially, when a batch falls back to solo for any entity, those **fallback solo prompts are also recorded** (`_record_solo_detail_prompt()` in both paths): the fallback calls are really sent to the LLM, so counting only the batch prompt would overstate the L2 saving exactly in the error path the safety design relies on. Because L2 *reduces the entity_detail call count by design*, the default per-call `matched-call-COUNT` scorer would drop every batched turn; `tools/ab_paired_score.py --per-turn` scores the **per-turn TOTAL** `raw_input_tokens` over the matched-TURN population instead, which is the fair, non-blind comparison for L2. In `--per-turn` mode `--noise-floor` is in tok/turn units and is **required** (no per-call default applies, so a stale tok/call floor cannot produce a misleading `SEPARABLE` verdict). The #488 paired-A/B gate still applies: validate with ≥3 runs/arm before flipping the default.

**Relevance-Detail Shadow** (S0, epic #477, lever (b)/A2a, #494): A **measurement-only** instrument that validates the relevance signal is sound *before* any behaviour-changing slice (S1) gates `entity_detail` selection on it. Today the per-turn detail set is whatever discovery qualified, truncated by the confidence-sorted cap `_MAX_DETAIL_ENTITIES_PER_TURN = 6` (`_entity_detail_capped_count` records how many were dropped), which silently drops real state updates on busy turns. S0 computes — **without changing the actual selection** — the set the existing relevance signal (`_select_context_aware_entities`, #233; mentioned → co-located → one-hop active relationship) WOULD pick, force-keeping new entities and the PC exactly as S1/A2a will, and logs the comparison. Gated via `context_optimizations.relevance_shadow_logging` in `config/llm.json` (**default OFF**). Implements **#494 S0**.

- **Zero behaviour change**: the shadow is computed in `extract_and_merge` strictly *after* the cap loop finalises `_entity_tasks`, reads only the catalogs and the (pre-cap / post-cap) task lists, and writes only a new `relevance_detail_shadow` key into the per-turn extraction-log record. It never changes which entities are detailed, the prompts, the LLM calls, or any extracted output. Flag-OFF and flag-ON produce **byte-identical** extraction results (catalogs, events) and an identical detail-call selection; only the extra log stream differs. When OFF the log record is byte-identical to the pre-#494 baseline (the key is omitted, not set to null). A unit test (`tests/test_relevance_detail_shadow.py::TestOutputIdentity`) runs `extract_and_merge` flag-ON vs flag-OFF over a real >cap turn and asserts the catalogs, events, and detail-call order are identical and that `relevance_detail_shadow` is the *only* added log key.
- **Logged metrics** (per turn, under `relevance_detail_shadow`): `actual_detailed_count`/`_ids` (the post-cap set actually detailed), `cap_dropped_count`/`_ids` (what the cap-6 dropped), `relevance_would_select_count`/`_ids` (what relevance would pick — bounded by scene size, ~constant, vs the cap-6 truncation), `cap_dropped_but_would_include_count`/`_ids` (the silently-dropped updates relevance recovers), `actual_detail_would_drop_count`/`_ids` and `mentioned_but_unchanged_fraction` (of the current detail calls, the relevance-droppable "mentioned-but-unchanged" fraction A2a expects to reclaim — a *pre-detail relevance proxy*, since S0 runs before detail extraction by design and takes no extra LLM pass), a `per_entity` record (one entry per pre-cap candidate: `id`, `would_select`, `reason` ∈ {`new`, `pc`, `mentioned`, `adjacent`, `none`}, `referenced`, `is_new`, `is_pc` — this is the per-entity relevance "score" S0 logs so each turn's selection is auditable entity-by-entity), and the **critical safety metric** `referenced_but_would_drop_count`/`_ids` — entities **referenced in the turn that relevance would drop**. A referenced entity being dropped is the coreference-safety failure mode, so this must be ~0 for the signal to be sound. It is a **real measurement, not zero-by-construction**: reference detection for this metric runs with the `_COMMON_WORD_BLOCKLIST` suppression **disabled** (`_find_mentioned_entities(..., apply_blocklist=False)`), so an entity literally named in the turn (e.g. one named "Shadow" or "Spirit") that relevance would drop is correctly counted rather than hidden by the selection-path blocklist. The detection is deliberately over-inclusive (false positives acceptable, false negatives are not) precisely so the go/no-go data cannot report a reassuring zero while missing a real referenced-but-dropped case.
- **Rule 9 / Rule 10**: the only new knob is the boolean flag `relevance_shadow_logging` (default OFF). The would-select rule reuses the existing #233 discrete tier membership and the new/PC force-keep floor — **no new scalar threshold and no domain word list**. The scalar `relevance_detail_threshold` named in the design is an S1 (gate) knob, deliberately *not* introduced by this measurement-only slice. `_MAX_DETAIL_ENTITIES_PER_TURN = 6` remains a safety ceiling, untouched.

**Relevance-Scoped Entity Detail** (S1a recover-only, epic #477, lever (b)/A2a, #498): The first **behaviour-changing** slice that wires the #233 relevance signal the S0 shadow validated into the actual `entity_detail` selection. When ON (`context_optimizations.relevance_scoped_detail.enabled = true`, **default OFF**, `mode = "recover_only"`), the detail set becomes the cap-6 set **UNION** the *relevance recoveries* — cap-dropped candidates the existing relevance signal (`_select_context_aware_entities`, #233; mentioned → co-located → one-hop active relationship, plus the new/PC floor) marks relevant. This recovers the real state updates the confidence-sorted cap-6 silently drops on busy turns (S0 measured ~245 such cap-dropped-but-relevant updates run-wide). Implements **#498 S1a**.

- **Recover-only — a superset, drops nothing**: S1a only *adds* recoveries to the cap-6 set; the result is always a **superset** of the cap-6 set, so no currently-detailed entity is ever dropped. (Dropping the off-scene filler the cap currently keeps is the separate S1b `mode = "drop_filler"` slice, **not** implemented here.) The selection runs in `_select_relevance_scoped_detail_recover()` immediately after the cap-6 loop and **before** `_partition_detail_tasks()`, so L2 batching composes unchanged on the selected set.
- **Bound (ceiling + floor)**: `_apply_relevance_ceiling()` bounds the set to `relevance_scoped_detail.max_detail_entities` (the **ceiling**, default **10**). The default is calibrated from S0 telemetry — `would_select` p95 = 10 (max = 18, p50 = 3) — so the ceiling caps the climax-scene spike while binding on < 5% of turns. On bind it trims the **lowest-relevance recovery tail** (ranked by the `_select_context_aware_entities` ordering — strictly better than the cap-6 lowest-confidence drop), never the cap-6 set and never the **floor** (PC, new, every mentioned entity). The floor is a hard lower bound that the ceiling may not violate, so a very busy scene can legitimately exceed the ceiling. The ceiling is clamped `>= _MAX_DETAIL_ENTITIES_PER_TURN = 6` so recover-only can never be forced to drop a cap-6 member.
- **Flag-OFF byte-identity (the A/B control)**: when the flag is absent or `false`, the new branch is never entered and the cap-6 `_entity_tasks` is used unchanged, producing **byte-identical** extraction output to the cap-6 baseline (`tests/test_relevance_scoped_detail.py::TestEndToEnd::test_flag_off_byte_identical`). The S0 shadow (`relevance_detail_shadow`) stays available during a S1 run so live `would_select` can be audited against the now relevance-driven actual set.
- **Validation gate**: S1a is gated on a **quality** update-retention A/B (recovered-update diff vs a cap-6 control), **not** the 60h paired-token A/B — by design its per-turn token delta is net ~flat-to-marginally-positive (it spends a few tokens to recover dropped updates), so the token metric would sit within run noise. The `calls/turn`-vs-catalog-size plateau is inherited from S0; the total-input-token slope is lever (c)'s (S2/S3) deliverable, out of scope here.
- **Rule 9 / Rule 10**: the only genuinely new tuning knob is the ceiling `max_detail_entities` (a config default, telemetry-derived from S0 p95, recalibrated from the `would_select` / ceiling-bind-rate histograms). Selection reuses the #233 discrete tier membership — **no new scalar relevance score, no domain word list**. The `mode` field (`recover_only` today; `drop_filler` reserved for S1b) lets the tighten slice land without a new flag.

**Raw-IO Capture** (step 1, epic #477): A **measurement-only** instrument that tees the **verbatim** LLM prompt + completion of every extraction call to a dedicated artifact, so a downstream analysis can verify — on real model output, not a parsed-catalog proxy — what each phase (and `entity_detail` in particular) actually emits. It exists to close the gap the round-53 "output is ~80% restatement" claim exposed: that analysis was computed from the **parsed catalog**, which conflates the `entity_detail` body (~16 schema fields) with the separate `relationship_mapper` phase; only the raw completion can settle what `entity_detail` truly restates. Gated via `context_optimizations.raw_io_capture` in `config/llm.json` (**default OFF**, strict bool `is True`). Implements **#477 step 1**.

- **What it captures**: when ON, the `LLMClient` (`tools/llm_client.py`) tees **one JSONL record per LLM call** to `<framework-dir>/raw-io-capture.jsonl` — kept **separate** from `extraction-log.jsonl` because the raw text is large. The record is teed **before** JSON parse / empty / truncation handling, so failed, truncated, parse-rejected, and retried completions are **all** captured (not just clean successes) — the measurement set is not biased toward successful parses. Each record is `{timestamp, turn, phase, entity_id, entity_ids, raw_prompt:{system,user}, raw_completion, input_tokens, output_tokens, input_tokens_estimated, output_tokens_estimated}`, serialised with `ensure_ascii=False` so non-ASCII prompt/completion text stays verbatim and human-readable. `phase` ∈ {`entity_detail`, `entity_detail_batch`, `entity_detail_backfill`, `entity_detail_refresh`, `discovery`, `relationship_mapper`, `event`} (`generate_text` callers may pass others, e.g. `synthesis`); `entity_id` is attached for the per-entity `entity_detail` calls (including the PC call, tagged `char-player`), so the **per-entity PC-call-isolated** input/output tokens are recoverable. For the **batched** `entity_detail_batch` call (`entity_id` is `None`, since it covers many entities), `entity_ids` lists every entity the single prompt presented, keeping the batched call per-entity attributable.
- **Output (decode) tokens — the never-before-logged 87%-wall-clock driver**: `output_tokens` is taken from the backend's **real** count in priority order: the Ollama native-streaming `eval_count` from the final stream frame, then the OpenAI-compatible `usage.completion_tokens` (`output_tokens_estimated: false`); only when neither is present does it fall back to a char-heuristic estimate of the raw completion (`_estimate_tokens`) flagged `output_tokens_estimated: true`. `input_tokens` follows the same rule via the streaming `prompt_eval_count` / `usage.prompt_tokens`. (A streaming count of `0` is treated as "not reported" and falls back to estimate.) Per-call input tokens already existed (#484) as a pre-compaction measure; this records the **actual sent** call's tokens per entity.
- **Where it's wired**: the capture point is the `LLMClient` boundary (`extract_json` / `generate_text`), which already holds the raw prompt + raw response + token counts for every call. The phase callers in `tools/semantic_extraction.py` pass a small `capture={"turn","phase","entity_id"[,"entity_ids"]}` tag through — including the **auxiliary** entity-detail paths `backfill_stubs()` (`entity_detail_backfill`) and `refresh_entities()` (`entity_detail_refresh`) so they are not measurement blind spots. The client writes the record only when capture is enabled **and** a tag is supplied. Capture also propagates to the **fallback** provider, so a completion served by fallback (after the primary exhausts retries) is captured too (the two clients share one append lock). The writer is **best-effort** (any serialisation/IO error is swallowed) so capture can never interrupt extraction.
- **Zero behaviour change, zero overhead when OFF**: the entry points (`extract_semantic_batch`, `extract_semantic_single`) call `llm.enable_raw_io_capture(path)` **only** when the flag is on. While disabled, `self._raw_io_capture_path is None`, so `extract_json`/`generate_text` do **no** capture work — no record built, serialised, or written — and **no artifact is created**. The `capture` kwarg never enters the API request body, so flag-OFF runs are **byte-identical** to baseline (`tests/test_raw_io_capture.py::TestFlagOff`). When ON it only **writes an extra artifact**: same prompts, same calls, same parsing, same catalog.
- **How it's used**: measurement-only — it needs **no GPU token A/B** to merge. It is exercised by **one flag-on extraction run** to produce the raw data the entity_detail-output analysis consumes (and which gates the later delta-output work). The PR trips the "A/B Test Required" path automation because it touches `semantic_extraction.py`/`llm_client.py`, but the validation is a flag-on capture run, **not** a token A/B.
- **Rule 9 / Rule 10**: the only new knob is the boolean flag `raw_io_capture` (default OFF) — no thresholds, no word lists. Raw prompts contain game-transcript text (not secrets); the capture never includes client config, API keys, or credentials.

**Relationship-Demotion Shadow** (step 1, epic #477, #495): A **measurement-only** instrument that logs *which* PC relationships a contemplated demotion policy **would** drop — without changing any extraction behaviour — so the real relationship-context lever (#495) can later be designed from per-turn telemetry. The PC (`char-player`) relationship block grows unbounded (~136 relationships by turn 344, ~44% stale beyond 100 turns, ~10k tokens — the single largest context block and dominant `relationship_mapper` OUTPUT-tail driver); before bounding it we need real data on which relationships are stale, how connected their endpoints are, and whether each is kept because it was genuinely mentioned this turn or only because it sits on the PC's injected floor. Gated via `context_optimizations.relationship_demotion_shadow` in `config/llm.json` (**default OFF**, strict bool `is True`). Implements **#495 step 1** (it does **not** close #495 — the behaviour change remains pending).

- **Two seams**: the shadow runs at **both** relationship-serialization points in `extract_and_merge`. **Seam A** = `entity_detail` prior-context (`_format_prior_entity_context`): the **RAW relationship list** (pre type-tier cap / pre scene-filter) of each entity detailed this turn — i.e. the unbounded **source** list, the PC block being the #495 driver. *Caveat:* with `relationship_type_tiering` on (the default) the set actually injected at seam A is a smaller already-capped subset (`_apply_pc_rel_type_tier` caps the PC block; non-PC entities are scene-filtered), so seam-A telemetry measures the **unbounded source**, not the injected block. **Seam B** = `relationship_mapper` existing-rels (`_collect_existing_relationships`): the mentioned entities' relationships. Each emits one record per relationship per seam per turn, labelled `seam` ∈ {`"A"`, `"B"`}.
- **Artifact + fields**: when ON, records are appended to `<framework-dir>/relationship-demotion-shadow.jsonl` — kept **separate** from `extraction-log.jsonl` because it is large (one line per relationship per seam per turn). Each record is `{turn, owner, target_id, type, status, staleness, core_degree, mention_source, would_demote, seam}`: `staleness` = `current_turn − last_updated_turn` (`null` if unparseable); `core_degree` = endpoint connectivity (`len(forward)+len(reverse)`) from the in-memory bidirectional relationship index (`generate_relationship_index`, built once per turn; `null` when the endpoint is absent); `mention_source` ∈ {`real_mention` (the rel's endpoint was genuinely named in the turn text — beats the floor), `injected_floor` (a `char-player` relationship kept only because it sits on the PC floor, not mentioned this turn), `none`}; `would_demote` (bool).
- **`would_demote` reuses the existing tier scoring (Rule 10)**: it is `(tier == 4)` from the **existing discrete tier-4-OMIT scoring** already used by seam B's budget trim, extracted into the shared `_score_relationship_tier()` helper that both seam B and the shadow call. It introduces **no new tuned scalar threshold** — the demotion policy is a documented measurement *assumption*, not shipped behaviour. *Methodology caveat (seam A):* seam A scores `would_demote` with seam B's tier logic over a **synthetic** mention set (`set(owners) | real_mentioned`), so it is structurally conservative and is **not** seam A's real `_apply_pc_rel_type_tier` policy. *Methodology caveat (budget):* `would_demote` is the **budget-independent pre-budget tier-4 set**; seam B's effective omit set under context-budget pressure (`_format_relationships_budgeted` degrading tier-3→omit) is larger, so `would_demote` **under-counts** what seam B currently omits late-run.
- **Zero behaviour change, zero overhead when OFF**: the shadow is computed strictly after the seam inputs are finalised, reads only the catalogs/relationship-index (never mutates a relationship, prompt, or scoring state), and writes only the separate artifact. When OFF (or no `framework_dir` sink is threaded), the gated block is never entered: **no record built, no artifact created, zero work** — flag-OFF extraction is **byte-identical** to baseline (`tests/test_relationship_demotion_shadow.py`). Flag-ON is byte-identical extraction with only the extra log stream.
- **How it's used**: measurement-only — exercised by **one flag-on extraction run** to produce the telemetry that informs the #495 behaviour-changing lever; it needs no token A/B to merge.
- **Rule 9 / Rule 10**: the only new knob is the boolean flag `relationship_demotion_shadow` (default OFF) — no thresholds, no word lists. The would-demote policy reuses the discrete tier-4 membership; no new scalar cutoff is shipped.

**Prompt Token Instrumentation**: Every extraction phase logs estimated input tokens in the extraction log (`prompt_metrics` field) for performance monitoring.

#### Raw-vs-Compressed Instrumentation (#464, PR-1)

Compression is **born observable**: raw-vs-compressed measurement landed *before* any adaptive behavior change so a regression can never again hide in a stderr-only print (the #460/#463 blind spot). PR-1 adds measurement only — the assembled prompt is byte-for-byte identical to the prior baseline, so `raw_input_tokens == input_tokens` and `compression_ratio == 1.0` on every successful extraction-log record (the faithful no-op). Quota-exhaustion and exception records are written without these fields.

Per-phase, under `prompt_metrics.<phase>` in `extraction-log.jsonl`:

- `raw_input_tokens` — pre-compression estimate. In PR-1 it defaults to the compressed size (no new compression layer is active). **Exception (#484): the `entity_detail` phase now passes a TRUE pre-compaction `raw_tokens`** — `format_detail_prompt(..., return_uncompacted=True)` surfaces the prompt built from the full, untrimmed prior entity state alongside the compacted prompt actually sent. So when the existing detail-prompt compaction (volatile digest/cap, relationship filtering/compaction) actually trims content, `raw_input_tokens > compressed_input_tokens` and `compression_ratio < 1.0` for `entity_detail`; when nothing is trimmed they remain equal (no false positive). This is measurement-only: the prompt sent to the model and the extraction output are unchanged. It unblocks the A/B of any prompt-compaction transform (e.g. PR #483's A0 checkpoint-compaction), which was previously invisible because `entity_detail` measured tokens *after* compaction.
  - **Guardrail (#485 review):** the uncompacted measurement adds one extra `json.dumps` of the full prior entity state per entity per turn. On long sessions with large catalog entries this serialisation can dominate per-turn CPU/memory. The config flag `entity_detail_raw_instrumentation` (default `true`) gates it; set it to `false` to skip the extra serialisation, in which case `entity_detail` `raw_tokens` falls back to the compressed size (the pre-#484 behaviour) and `compression_ratio` returns to `1.0` for that phase. Token estimation itself stays cheap (`len(text)//3`), so the only added cost is the gated serialisation.
- `compressed_input_tokens` — final assembled prompt size (mirrors `input_tokens`).
- `compression_ratio` — `compressed / raw` (`1.0` in PR-1).
- Section drop counters: `catalog_entries_pruned`, `catalog_entries_degraded`, `relationships_pruned`, `volatile_snapshots_dropped`, `turns_dropped` (all `0` in PR-1).

Per-turn, two new top-level blocks:

- `turn_compression` — `raw_input_tokens_total`, `compressed_input_tokens_total`, `compression_ratio_total`, and `activated_phases` (empty in PR-1). A phase is "activated" only when its ratio drops below 1.0.
- `compression` — the quality-observability signals: `dropped_entity_ids[]`, `dropped_entity_count` (by type), `dropped_relationship_count`, `dropped_event_ids[]`/`dropped_event_count`, `dropped_plot_thread_ids[]`, `context_window_turn_floor`, `included_vs_total`, `dropped_then_referenced[]` (the dropped-but-surfaced join key), plus `strategy` and `params`. In PR-1 the dropped lists are empty, `included == total`, `strategy == "baseline"`, and `params == {}`.

Per-turn stderr lines replace the former lone `[ctx-opt]` print so raw-vs-compressed and the discovery-floor guardrail are continuously visible:

```
[COMPRESSION] turn-072 raw=27310 comp=14880 ratio=0.54 phases=entity_detail,relationship_mapper (ACTIVE)
[RETENTION]   turn-072 detail: pruned=0 degraded=11 rel_pruned=14 vol_dropped=9 | discovery: floor_held=yes omitted=0 discovered=8
```

In PR-1 these print with `ratio=1.00`, `(INACTIVE)`, all drop counters `0`, `floor_held=yes`, and `discovered=N` (the count of LLM-discovered entities for the turn, not a priority-retention ratio).

The three compression surfaces (`format_known_entities_bounded`, `_collect_existing_relationships`, `_trim_entry_for_scene`) gained backward-compatible `return_stats` paths; the stats payload differs per surface: `format_known_entities_bounded` reports `raw_tokens` plus entity-level counters; `_collect_existing_relationships` reports both pre- and post-compression token estimates plus tier counters; `_trim_entry_for_scene` reports drop counters only. PR-1 leaves the call sites on the result-only return (no behavior change); PR-2 wires these stats into the per-turn metrics.

**Aggregator**: `tools/agg_compression.py` (successor to the ad-hoc `_agg_pr463.py`) reads the new fields and buckets metrics by turn-index band (1-20, 21-50, 51-100, 101+), emitting a per-band raw/compressed/ratio table. Turn-band bucketing is the report that gates PR-2 — the late-vs-early split is exactly what hid the #393/#394/#463 regressions when only session totals were checked.

**Paired A/B scorer**: `tools/ab_paired_score.py` (#487) judges a candidate's effect on the pre-compaction `entity_detail` token metric without trusting any single run. The full-depth A==A determinism test proved the pipeline is non-deterministic (84/344 turns diverge on `new_entities`, 51/344 on `new_events`), so the single-pass byte-diff is deprecated for this metric. The scorer takes N=2–3 runs per variant, filters to **matched-call turns** (turns where every run made the same number of `entity_detail` calls so the per-call metric is comparable), computes paired B−A deltas, and reports the weighted Δ, per-turn mean/median Δ, and Cohen's d **against the measured ~5 tok/call weighted noise floor**. See [ab-test-standard.md](ab-test-standard.md) §0.

### Timeline Layer (Framework)

Estimated timeline of in-game events, anchored to a configurable reference point.

- `framework/catalogs/timeline.json` — temporal markers with day estimates, seasons, and confidence scores
- `framework/catalogs/timeline.md` — narrative timeline wiki page (auto-generated)
- `tools/temporal_extraction.py` — pattern-based extraction of season keywords, biological markers, construction milestones, and time-skip language
- `templates/extraction/temporal-signals.md` — optional LLM prompt template for ambiguous temporal estimation
- Calibrated from biological markers (pregnancies), construction timelines, and seasonal descriptions
- Reference anchor defaults to turn-001 (Day 0); a named anchor (e.g., settlement founding) can be set in config
- Feeds into wiki page generation for character ages and event dating (estimated day column in event timelines, season labels in infoboxes)
- **Season flicker filtering** (#275, #277): Low-confidence season signals are filtered using a sliding-window approach to prevent noise from regex false positives (e.g., "summer" keywords appearing in a winter-dominant story). A signal is kept only if its confidence ≥ threshold (default 0.6) OR at least `min_support` (default 1) neighboring season entries within a per-side radius of `window_size` (default 5) share the same base season. Non-season entries are never filtered. Base season detection itself requires at least 2 distinct keyword pattern matches and a margin of 2 over the runner-up to avoid false positives from single keyword occurrences.
- **Anchor event detection** (#275): The system auto-detects the most significant temporal anchor from timeline data — preferring explicit `anchor_event` type entries, then falling back to the first `time_skip` or `biological_marker`.
- **Narrative timeline wiki page** (#275, #278): `generate_wiki_pages.py --type timeline` (or default full generation) produces `timeline.md` with: (1) a Current Position infobox showing estimated day, season, anchor, and confidence; (2) a structured narrative summary with story progression beats (when `events.json` is available) or a concise fallback; (3) reference tables for season progression, time passages, biological markers, and other milestones. The narrative groups biological markers into pregnancy-birth cycles and uses event catalog data for period-based story summaries.

## Architecture Principles

### No Hardcoded Word Lists in Programmatic Filters

Entity filtering MUST NOT use hardcoded lists of specific words (body parts, common nouns, game terms, combat actions, elements, role names, etc.) in Python code. This approach is:

- **Not scalable** — every new edge case requires adding more words (whack-a-mole)
- **Not semantically grounded** — a flat word list cannot distinguish context-dependent meanings
- **A maintenance burden** — lists grow indefinitely and interact unpredictably

The LLM extraction templates (`templates/extraction/*.md`) are the correct place to define what constitutes a valid entity. Templates have semantic understanding and can apply nuanced classification rules that word lists cannot.

**Acceptable inline constants:**

- Function words (articles and prepositions) used for tokenization: `a`, `an`, `the`, `of`
- Pronoun stems used for ID generation: `he`, `she`, `they`, `it`

**NOT acceptable as inline constants:**

- Domain-specific word lists: body parts, combat terms, elements, role names, generic nouns, abstract concepts
- Any set of >5 domain-specific strings used for entity rejection

**Policy:**

- Existing hardcoded word lists in the codebase are technical debt to be migrated to template-based filtering
- New PRs that add hardcoded domain-specific word lists for entity filtering MUST be rejected during review

---

### Source-Quality First (Prompt and Model Before Heuristics)

When extraction quality is inadequate — missing entities, duplicates, or noise — the first remedy is a **prompt-template change or model selection**, not a new post-processing pass. The post-processing heuristics in the extraction pipeline (orphan thresholds, stale-item sweep, staleness trimming, mention blocklists) are useful safety nets, but each new one is:

- **Fragile** — magic thresholds and sweep windows are calibrated to one model/campaign and silently degrade when either changes
- **Risky** — aggressive filters and sweeps can remove valid entities (over-deduplication, premature pruning)
- **Compounding** — heuristics interact unpredictably, making the pipeline harder to reason about

**Policy:**

- Fix quality at the source (template or model) whenever feasible; add a post-processing heuristic only when a source fix is infeasible.
- A new threshold or sweep parameter MUST be justified, documented here, and validated via an A/B entity-retention diff (#448). Recalibrate when the model changes.
- Evidence: in the deduplication experiment the coreference-template fix (#443) delivered the dominant gain, while the Python heuristics (cross-catalog gate, interval tuning) added no measurable benefit and risked over-deduplication. The smart-compression regression (#394) and stale-sweep regression (#441) are further cautionary cases.

This principle extends *No Hardcoded Word Lists* from word lists to thresholds and sweep parameters (see #413, #447).

#### Cross-cutting findings

1. **Pervasive staleness-window assumption (20 / 25 / 50 turns).** Five independent passes encode turn-windows sized for a 9B model and small catalogs. On 100–344-turn runs with Qwen3.6-35B these compound: an entity dropped from discovery → not re-extracted → goes stale → swept. They must be recalibrated together, not in isolation.
2. **No global entity-count floor.** Every degradation pass trims from the tail with only a token check; nothing guarantees a minimum number of entities survives. This is the structural gap behind #394 and #441.
3. **Hardcoded domain word lists** — `_COMMON_WORD_BLOCKLIST`, dedup `STOPWORDS`, `_GENERIC_STEMS`, and `_NON_LOCATION_NAMES` — are Rule 9 debt (overlaps #413; see *No Hardcoded Word Lists* above). Migrate them to template-based filtering rather than extending them.

**Phase 2 (recalibration) is deferred.** It depends on model characterization from #324 and must be validated empirically via the entity-retention A/B diff (#448): recalibrate the staleness windows and the detail-entity cap together against Qwen3.6-35B rather than tuning any single constant alone.

## Design Decisions

### Why file-based?

File-based storage is human-readable, git-friendly, and easy to edit manually or with Copilot. No database required.

### Why separate raw from derived?

Raw files are the source of truth. If analysis is wrong, derived files can be regenerated without loss.

### Why JSON schemas?

Schemas enable validation, documentation, and Copilot-assisted editing with type safety.

### Why no game-system assumptions?

The system learns the game world from the transcript. It works for any AI DM, any genre, any ruleset.
