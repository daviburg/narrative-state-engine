# Flag-OFF byte-identity golden — provenance

`flag_off_prior.json` pins the Phase A0 **flag-OFF** prior-state render
(`semantic_extraction._format_prior_entity_context`, config OFF) to byte-identity
with the pre-A0 `main` baseline — the A/B control for
`tests/test_checkpoint_compaction.py::TestFlagOffGolden`.

## Pinned base commit

- **Repo:** `daviburg/narrative-state-engine` (public)
- **Base ref:** `origin/main`
- **Base commit:** `5f0a38659f8055f32d07e9c6da6e5ac8f13c0246`
  (`feat: enable relationship_type_tiering by default (#481)`) — the merge-base
  of the A0 branch `feat/issue-482-a0-checkpoint-compaction` with `main`.

This is the exact commit the flag-OFF path MUST remain byte-identical to. The
A0 PR (#483) introduces NO change to the flag-OFF
`_format_prior_entity_context` code path.

## Regenerating command

The golden is regenerated by running the generator **from the pinned base
commit** (or any branch where the flag-OFF path is unchanged):

```sh
python tests/gen_a0_golden.py
```

## Why a base-branch subprocess guard also exists

A golden literal committed by the same PR that it guards is self-referential: a
regenerated fixture would still pass without ever executing the base-branch code.
`tests/test_checkpoint_compaction.py::TestFlagOffBaseBranchGuard` closes that gap
by materializing `tools/` **from the pinned base commit above** via `git archive`
and computing the flag-OFF renders in a subprocess loaded from that base code,
then asserting byte-identity with both this golden and the PR's flag-OFF output.
That guard CANNOT be satisfied by regenerating the fixture from the changed code.
